# OrdSA Schema

This directory holds the **canonical OrdSA construct schema** and **example deployment schemas** — the artifacts that make OrdSA machine-validatable.

The schema is the source of truth per [ADR-ORDSA-0001](../decisions/ADR-ORDSA-0001-schema-first-canonical-form.md). All other representations — the prose docs in [`../docs/`](../docs/), future ArchiMate models, future SysML v2 source, future UAF mappings — are *renderings* derived from or consistent with the schema, never the source.

## Contents

| File | Role | Mutability |
|---|---|---|
| [`ordsa-0.2.yaml`](ordsa-0.2.yaml) | **Construct schema** — the OrdSA construct itself for v0.2 | Immutable within version; changes via ADR |
| [`example-deployment.yaml`](example-deployment.yaml) | **Deployment example** — illustrative deployment that validates against `ordsa-0.2.yaml` | Example only; copy and modify in your own repo |

When v0.2 stabilizes and OrdSA cuts a v1.0 release, the construct schema moves to `ordsa-1.0.yaml`. Pre-1.0 minor versions (0.2, 0.3, …) may refine field names, allowed values, and conformance rules.

## The two-schema shape

Per [ADR-ORDSA-0003](../decisions/ADR-ORDSA-0003-construct-vs-deployment-schemas.md), OrdSA distinguishes two schema shapes:

### Construct schema (`ordsa-N.M.yaml`)

The **definition of the OrdSA construct itself** — the seven layers, allowed relationships, execution rights, conformance assertion vocabulary. Maintained by OrdSA via ADRs; immutable within a version.

Adopters do not edit this file. They cite it (`ordsa_construct: "0.2"` in their deployment) and validate against it.

### Deployment schema (`<your-deployment>.yaml`)

A **specific organization's claim of OrdSA conformance**. Names the deployment's concrete artifacts, orchestrators, agents, tools, substrates, and outcome stores; declares which conformance assertions it satisfies and how.

Each adopter authors their own deployment file, in their own repository, on their own lifecycle. The deployment file pins to a specific construct version via the `ordsa_construct` field.

## Validation flow (planned for v0.2 reference tooling)

```
deployment.yaml ──┐
                  ├──► ordsa-validator ──► verdict (pass / fail with reasons)
ordsa-0.2.yaml ─────┘                  │
                                     └──► structured report
                                          (assertion-by-assertion)
```

The reference validator ([ADR-ORDSA-0004](../decisions/ADR-ORDSA-0004-reference-tooling-python-pydantic.md): Python + pydantic) is **not yet implemented**. The schema itself is sufficient for human review; mechanical validation lands in a follow-up PR.

Until then, the schema serves three purposes:

1. **Documentation as data.** Reading [`ordsa-0.2.yaml`](ordsa-0.2.yaml) gives you the precise OrdSA semantics — what relationships are allowed, what rights exist, what conformance requires — in one structured file instead of scattered prose.
2. **Authoring target for deployments.** Adopters can start building their deployment file by copying [`example-deployment.yaml`](example-deployment.yaml) and editing it for their context.
3. **Foundation for the future tooling.** The validator, the JSON Schema emitter, the ArchiMate renderer ([ADR-ORDSA-0005](../decisions/ADR-ORDSA-0005-archi-as-canonical-modeling-tool.md)), and language bindings all consume this file as input.

## Versioning rules

- **Construct schema versions track OrdSA releases.** `ordsa-0.2.yaml` is the v0.2 construct; `ordsa-0.3.yaml` will be the v0.3 construct (if minor changes warrant a bump); `ordsa-1.0.yaml` is the first stable release.
- **Within a version, the construct schema is immutable.** Typo fixes and clarification of comments are acceptable; substantive changes (new layers, new execution rights, redefined relationships) require an ADR and ship in a new version.
- **Deployment schemas follow the deployment's lifecycle**, not OrdSA's. An adopter can update their deployment file as their AI architecture evolves without waiting on an OrdSA release.
- **Multi-version coexistence is intentional.** An organization may run deployments at different OrdSA construct versions. The validator handles all currently-supported versions.

## What's NOT yet here (and where it's going)

| Artifact | Status | Source |
|---|---|---|
| `ordsa-0.2.schema.json` (auto-emitted JSON Schema) | Not yet | Generated by reference tooling per ADR-ORDSA-0002 |
| `ordsa-validator` CLI tool | Not yet | Will live under `tools/ordsa-validator/` per ADR-ORDSA-0004 |
| ArchiMate XML rendering | Not yet | Will be a build-time artifact generated via pyArchimate subprocess per ADR-ORDSA-0005 |
| SysML v2 textual rendering | Future | Deferred to v0.3+ |
| UAF mapping appendix | Future | Deferred to v0.3+ |

Each missing item has a governing ADR; the schema is the input every one of them consumes.

## How to cite the schema

When an adopter publishes a deployment file:

```yaml
ordsa_construct: "0.2"
# URL form for external citations:
# https://github.com/osa-ai-org/ordsa-ai/blob/v0.2/schema/ordsa-0.2.yaml
```

When OrdSA releases are tagged in git (`v0.2`, `v1.0`, etc.), the URL above pins to the immutable schema at that point in time. Adopters should cite the tagged URL, not the branch URL, for stability.

## Feedback on the schema

Schema-level questions or proposed changes:

- **Question** — open a [Construct Q&A discussion](https://github.com/osa-ai-org/ordsa-ai/discussions) before authoring an ADR
- **Proposed construct change** — author an ADR per [`../CONTRIBUTING.md`](../CONTRIBUTING.md), label the PR `adr`
- **Bug in the schema** (typo, broken cross-reference, internal inconsistency) — open an Issue
